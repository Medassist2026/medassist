Data Flow — Where Each Section Gets Its Data
🗄️ Database Layer — Supabase Tables
📋
queue
Walk-in patients checked in by the frontdesk. Each row = a patient physically present at the clinic right now. Frontdesk adds them via the queue management screen. Includes: patient_id, doctor_id, checked_in_at, status, priority
→ في الانتظار الآن → عداد الطابور
📅
appointments
Pre-scheduled bookings with a specific time slot. Filtered by doctor_id = me AND date = today. Includes: patient_id, start_time, duration_minutes, status, appointment_type
→ مواعيد اليوم → عداد المواعيد → عداد المكتمل
👤
patients
Patient profile data. Joined into both queue and appointments queries to get name, age, sex, phone.
→ joined into both
🏥
doctor_availability + clinic_memberships
Used for doctor name, clinic name, and working hours shown in header and badge.
→ header / greeting
↓ API Routes
⚡ API Layer — Next.js Route Handlers
🔴
GET /api/doctor/queue
Returns active queue for today: status IN ('waiting','called'), joined with patients. Sorted by priority DESC, checked_in_at ASC. Includes wait time in minutes.
feeds → في الانتظار الآن
🔵
GET /api/doctor/appointments?date=today
Returns today's appointments sorted by start_time ASC. Already exists — just needs filtering to today only and correct timezone.
feeds → مواعيد اليوم feeds → عداد المكتمل
↓ UI Components
🖥️ UI Layer — Dashboard Sections
🟠
في الانتظار الآن (Live Queue)
Patient cards sorted by wait time. Shows name, age, wait duration, priority badge. "بدء الجلسة" starts a session immediately. Refreshes every 30 seconds (polling) or via realtime subscription.
source: queue table live / realtime
🔵
مواعيد اليوم (Today's Timeline)
Compact time-sorted list. Status dots: yellow = arrived but not in queue yet, blue = upcoming, green = done. When frontdesk checks in an appointment holder, that appointment's status becomes "arrived" — it moves up, but stays in this section (not promoted to queue until doctor calls them).
source: appointments table date = today

⚡ Key rule: "في الانتظار الآن" on the doctor's dashboard = queue rows only. "مواعيد اليوم" = appointments rows. Never mix. An appointment patient crosses into the queue only after the frontdesk confirms they have physically arrived.
Design Decisions Explained
1
3 separate counters, not 1
Current dashboard shows one merged "مرضى اليوم: 2" that mixes queue + appointments. Splitting into Waiting Now / Scheduled Today / Completed gives the doctor instant situational awareness without reading cards.
queue.count(status='waiting') + appointments.count(date=today) + appointments.count(status='completed')
2
Wait time on queue cards
Each queue card shows how long the patient has been waiting (now - checked_in_at). Color-codes to orange/red as wait time grows. This is not possible with appointments because they haven't arrived yet.
wait_minutes = floor((now - checked_in_at) / 60)
3
Appointments stay informational
The appointment timeline section is read-only for the doctor. It tells them what's coming. The dot color changes from blue → yellow when the frontdesk marks the patient as "arrived" — but they only enter the action queue when the doctor is ready to call them.
yellow dot = status='arrived', blue = status='scheduled'
4
Realtime vs. polling
Queue section uses Supabase Realtime subscription (supabase.channel('queue')) so new walk-ins appear instantly without refresh. Appointments section is fine with a 60-second poll since those don't change frequently.
supabase.channel('queue').on('postgres_changes', ...)
