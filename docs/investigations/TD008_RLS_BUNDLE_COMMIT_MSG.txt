1. Offline-write Phase 1 (TD-008, D-050) — migration 069
================================================================

Resolves TD-008. After D-043/D-044 cleanup, idb-cache.ts had the
offline-write primitives but no write surface enqueued. This commit
wires the three high-frequency frontdesk surfaces (check-in, payment
create, clinical-note save) onto the queue with replay-safe
idempotency.

Storage unification
-------------------
useOfflineMutation hook refactored from localStorage onto
idb-cache.addPendingWrite / syncPendingWrites / getPendingWriteCount.
Same hook API for components, different backend. One-shot legacy-
localStorage drain runs on first hook mount post-deploy so any
clinic that was offline at deploy time does not lose pending writes.

Server idempotency by surface
-----------------------------
- Check-in: natural dedupe key already exists (patient_id +
  doctor_id + Cairo-day-anchored "today"). Handler now returns 200
  with `deduped: true` and the existing record instead of the
  previous 409. The client surface treats deduped:true as an inline
  banner instead of a hard error.
- Payments + clinical_notes: no natural dedupe (a doctor can write
  two notes in a visit; a patient can pay cash twice). Migration 069
  adds `client_idempotency_key TEXT` with a partial unique index
  `WHERE client_idempotency_key IS NOT NULL` on both tables. Client
  surfaces (payments/new, SessionForm) generate nanoid() per submit;
  handlers look up by key first, return existing record on hit.
- Belt-and-suspenders: idb-cache.syncPendingWrites treats both 2xx
  and 409 as success on replay, so any older pre-this-commit client
  still in production also flushes correctly.

Test discipline
---------------
New packages/shared/hooks/__tests__/useOfflineMutation.test.ts
follows the existing compile-time-witness pattern (same as
frontdesk/payments/create/__tests__/handler.test.ts). Locks the
hook surface, the clientIdempotencyKey-as-optional data-layer
contract, and the idb-cache 409-as-success invariant.
