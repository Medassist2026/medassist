2. Run `audits/REVIEW_CRITERIA.md` §1 pre-work verification against the prompt — confirm empirical claims hold against ground truth, STATE_OF_WORK.md is current with HEAD, and cross-references resolve. Surface contradictions BEFORE authoring.
3. Pick up the active workstream's Next Action OR start the highest-priority pending workstream.
4. Update `STATE_OF_WORK.md` "Currently doing" before starting work.
5. At session end (whether work is done or paused):
   - Update active workstream "Status", "Currently doing", "Next action", "Last commit".
   - Update "Last updated" + "Last session" header.
   - Update the 3 core docs (`ARCHITECTURE.md`, `DECISIONS_LOG.md`, `PRODUCT_SPEC.md`) for any architectural / decision / product-scope changes shipped this session — even if a single sentence.
   - Update `audits/REVIEW_CRITERIA.md` if a new failure-mode pattern surfaced this session OR Mo's feedback contradicted an existing criterion. The doc is part of the lockstep set.
   - Commit `STATE_OF_WORK.md` + the 3 core docs (+ REVIEW_CRITERIA.md if changed) together.
6. Surface session-end report referencing `STATE_OF_WORK.md`, not project memory. Apply REVIEW_CRITERIA.md §3 surface format requirements — required elements, conventions, what NOT to include.

### Prompt template (standing requirement)

Every workstream prompt henceforth includes a "Pre-work verification" reference: cowork session verifies the prompt's empirical claims against ground truth BEFORE starting work, per `audits/REVIEW_CRITERIA.md` §1.1. Contradictions surface as a STOP (never silent execution). The prompt is input data, not infallible spec — REVIEW_CRITERIA.md §5 grants the cowork session explicit authority to push back on Mo's prompts when premises are wrong, scope is internally inconsistent, or the verification methodology won't actually verify what it claims to verify.
