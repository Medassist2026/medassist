# Patient Phone Duplicate Resolution

> Source of truth for the canonical/loser mapping that mig 071 backfills
> into `patients.is_canonical` and `patients.duplicate_of_patient_id`.
> Generated by the BUILD prompt that ships mig 071/072.
>
> Resolution rule (locked): **oldest `patients.created_at` wins; ties
> broken by `patients.id` ascending (deterministic).** No row is
> deleted. No FK is repointed in this prompt; that work belongs to
> Prompt 3 (data layer cutover).
>
> Cross-clinic duplicates are EXPECTED. A phone appearing once per
> clinic is the desired pattern post-rollout — one global row, one
> `patient_clinic_records` row per clinic. The dedup flag chooses the
> seed row that becomes the global_patients row in mig 072. Per-clinic
> bookkeeping is then re-derived from the loser rows by mig 074
> (Prompt 4, out of scope here).

The migration is structured so this works without re-deployment:

1. Mig 071 lands on staging (phone normalization only).
2. Mig 072 lands on staging (dedup detection + auto-population of
   `_patient_dedup_plan`). Mo runs the **detection script** (below)
   and pastes the output into "Cluster inventory" + the
   "_patient_dedup_plan inventory" section of this file.
3. Mo reviews:
   - Auto-rule rows (`resolution = 'auto_oldest_wins'`) need no
     intervention — they have `decided_at = NOW()` already.
   - Manual-review rows (`resolution = 'manual_review'`) have
     `decided_at IS NULL` and BLOCK mig 073 until updated. Mo runs
     the override SQL (next section) for each manual_review cluster.
   - If **count > 100**: STOP. Reopen the prompt with Mo for guidance.
4. Mo signs off this file. Mig 073 ships and consumes the plan.

The persistence in `_patient_dedup_plan` (072.4) is fully deterministic
from data in the duplicate views; the **decision authority** lives in
this file (Mo's review and overrides). Mig 073 is the consumer.

## How to override an auto-resolution

To change the canonical row for a cluster before mig 073 runs:

```sql
UPDATE public._patient_dedup_plan
   SET winner_patient_id = '<new-canonical-uuid>',
       loser_patient_ids = ARRAY[<former-canonical-uuid>, <other-loser-uuid>]::UUID[],
       resolution = 'manual_review',
       decided_by = '<your-auth-user-id>',
       decided_at = NOW(),
       notes = 'Reasoning: <why this row should win — e.g. "loser has all clinical_notes; winner is empty walk-in shell">'
 WHERE normalized_phone = '<phone>';
```

To accept a manual_review cluster without changing the placeholder
winner (acknowledge that "oldest wins" is correct here too):

```sql
UPDATE public._patient_dedup_plan
   SET decided_by = '<your-auth-user-id>',
       decided_at = NOW(),
       notes = 'Accepted auto placeholder: oldest wins; reviewed cross-clinic spread.'
 WHERE normalized_phone = '<phone>';
```

Mig 073's pre-flight assertion (073.1) refuses to run while ANY row
has `decided_at IS NULL`. That is the gate working as intended.

## Detection script (run after mig 072 applies)

```sql
-- Quick count (Phase A.A3 of the BUILD prompt):
SELECT COUNT(*) AS duplicate_clusters
  FROM public._patient_phone_duplicates;

-- Full inventory (paste output into Cluster inventory section below):
SELECT
  normalized_phone,
  dup_count,
  patient_ids,
  clinic_ids,
  full_names,
  earliest_created,
  latest_created
FROM public._patient_phone_duplicates
ORDER BY dup_count DESC, normalized_phone ASC;

-- Cross-clinic vs same-clinic split (the difference matters
-- for messaging consent re-derivation in Prompt 4):
SELECT
  CASE
    WHEN array_length(array_remove(clinic_ids, NULL), 1) =
         (SELECT COUNT(DISTINCT c) FROM unnest(clinic_ids) c WHERE c IS NOT NULL)
    THEN 'cross_clinic'
    ELSE 'within_clinic'
  END AS pattern,
  COUNT(*) AS clusters
FROM public._patient_phone_duplicates
GROUP BY 1;

-- _patient_dedup_plan inventory (decisions surface, populated by mig 072.4):
SELECT
  normalized_phone,
  resolution,
  winner_patient_id,
  loser_patient_ids,
  decided_by,
  decided_at,
  notes
FROM public._patient_dedup_plan
ORDER BY (decided_at IS NULL) DESC, resolution, normalized_phone;
-- decided_at IS NULL rows MUST be resolved before mig 073 runs.
```

## Manual-review queue

Only clusters that fail the auto-rule heuristics get listed here.
Heuristics that trigger manual review:

- Cluster size 3+ AND winner is a walk-in row with no clinical activity
  while a loser has clinical_notes — promote the loser (preserves data
  attribution).
- Two rows with identical `created_at` to the millisecond AND different
  `full_name` AND different `clinic_id` — likely a true two-people-one-phone
  case that needs human eyes.
- Winner row has `account_status = 'merged'` — pick the next-oldest.

## Quarantine review (rows with NULL normalized_phone)

Detection:

```sql
SELECT COUNT(*) FROM public._phone_normalize_quarantine;

SELECT q.row_id, q.raw_phone, p.clinic_id, p.full_name, p.created_at
  FROM public._phone_normalize_quarantine q
  LEFT JOIN public.patients p ON p.id = q.row_id
 WHERE q.table_name = 'patients'
 ORDER BY p.created_at ASC;
```

decision before mig 072 ships:

  via the existing patient-dedup admin endpoint.
