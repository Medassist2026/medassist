## Key Findings & Surprises

1. **No RESTRICTIVE policies**: All policies use PERMISSIVE. This is Phase A state—RESTRICTIVE may be added in Phase B.

2. **Three DENY-ALL placeholders exist**: `global_patients`, `patient_clinic_records` (legacy, never used), and intentionally-blocked tables (`patient_data_shares`, `patient_privacy_codes`, `privacy_code_attempts`, `privacy_code_sms_tokens`). These are by design.

3. **All live pv grants point to OWNERs**: Per memory notes, clinic_id logic always routes through clinic_memberships with role='OWNER' as the grantor. This is the "clinics are effectively solo" insight—every clinic has exactly one owner, and that owner controls data visibility.

4. **Three helper functions in active use**:
   - `is_clinic_member()` - checks clinic_id + clinic_memberships status='ACTIVE'
   - `can_access_patient()` - role-based patient access (READ, WRITE, etc.)
   - `can_open_messaging_conversation()` - gates messaging between doctor-patient

5. **Messaging is conversation-scoped**: Messages table doesn't directly check doctor_id/patient_id. Instead, it joins via conversations(id).

6. **Service role has backdoor access**: audit_log, sms_reminders, prescription_items all allow service_role full access (auth.role()='service_role'::text). These are safe edge function escape hatches.

7. **Registration tables allow auth.uid() self-insert**: users, doctors, patients, front_desk_staff all allow INSERT via `with_check: auth.uid() = id`. This is expected for registration flows.

8. **Clinic-scoped access is the dominant pattern**: 20+ tables use clinic_id + is_clinic_member() or can_access_patient() for role enforcement. This is the primary axis of data visibility in MedAssist.

9. **Doctor-patient relationship table is lightweight**: doctor_patient_relationships does NOT use clinic_id. It's a direct relationship store used for "active" checks (dpr.status='active'::text). This is independent of clinic membership.

10. **Lab results use FK-based access control**: lab_results doesn't have doctor_id/patient_id. It controls access via lab_orders(id) FK, checking lab_orders.doctor_id, lab_orders.patient_id, lab_orders.clinic_id.

## Gaps & Concerns for Phase B

- No UPDATE policies on payments (doctors can't update their own payments)
- DELETE coverage sparse (appointments, clinical_notes missing DELETE; payments, messages missing DELETE)
- Anonymous visits have duplicate "Doctors see own" policies (one with ALL, one with ALL—likely unintended duplication)
- Several tables missing INSERT when they have SELECT (lab_orders, prescription_items need UPDATE?)
- No explicit audit trail for RLS-blocked queries (DENY-ALL tables silently fail—may need logging in Phase B)
